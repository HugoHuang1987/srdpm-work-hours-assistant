## 二、SRDPM API 技术文档

### 认证方式
- Cookie `moka_s-rdpm_token` + Header `accesstoken`

### 接口列表
| 接口 | URL | 方法 | 参数 |
|------|-----|------|------|
| 审批列表 | `/srdpm-api/workload/approve/list` | POST | page, page_size, type, status, time, user_id, group_id |
| 统计 | `/srdpm-api/workload/approve/statistics` | POST | - |
| 明细 | `/srdpm-api/workload/approve/detail` | POST | - |
| 用户列表 | `/srdpm-api/workload/approve/userList` | POST | - |
| 审批通过 | `/srdpm-api/workload/approve/approval` | POST | `{"approve_id": "id1,id2,id3"}`（逗号分隔） |
| 驳回 | `/srdpm-api/workload/approve/refuse` | POST | `{"approve_id": "xxx", "note": "驳回原因"}` |

### 关键约束
- **审批单向**：待审→通过 不可撤回
- **驳回限制**：只能驳回待审状态，已通过不可逆
- **系统冻结**：每月7日冻结上月工时申报
- **查询状态**：status=0 返回待审，status=2/3 有数据，status=1 返回空
- **重要教训**：批量审批前必须先跑审核报告，区分正常/异常项，只审批正常项，异常项留人工处理

## 三、工时审批规则

1. **分工规则**：基于 Confluence 页面 22824730 附件「团队成员项目负荷_新拆分.xlsx」中的 A列(客户)、D列(机芯)、M列(BSP负责人)、O列(DIAG负责人)
2. **工时规则**：每人每天申报不超过打卡工时，90%比例最佳；**申报超标 = (总申报 - 休假/出差小时) / 打卡**
3. **漏报规则**：14人看谁没报或远低于打卡（<70%）
4. **组长规则**：
   - 梁郁沛：任意报
   - 何文亮：不报 G 的
   - 田昭辉：只报 G 的
   - **平台类（YF-CP/SW/纯平台类）不限组长规则**
5. **出差=请假**：title/content 含"出差"的无打卡不判异常，单独列出
6. **平台/公共事务**：type=纯平台类 或 items 以 YF-CP/YF-SW 开头，不判项目归属异常，但需单独汇总（按人+日期+内容+时长）

| 四 | 项目归属异常 | 需人工审核 (manual_approve) | 申报了不属于自己的项目 |

| 七 | 其他待定 | 纯信息展示 | 无法归入1-6的条目 |

### 5.2 状态系统（2026-07-10 最终版 — 本轮对话核心成果）

**三态设计**：
```
pending → confirming → approved
(未标记)   (本地已标记)   (SRDPM真实审批)
```

| 状态 | 来源 | 含义 | 显示 |
|------|------|------|------|
| `pending` | 默认 | 未标记 | 三/四显示"待审核"，二/五/六显示"未审批" |
| `confirming` | localStorage | 本地已标记，**SRDPM尚未审批** | 三/四显示"已标记"+"撤销标记"按钮，二/五/六显示"已勾选"+"✓已勾选"按钮 |
| `approved` | SRDPM JSON (item.status==="approved") | SRDPM已真实审批，**不可逆** | 所有分类显示"已审核 ✓" |

**关键规则**：
- `localStorage` 只能存 `confirming` 或 `pending`，**不能存 `approved`**
- `approved` 状态仅来自 SRDPM 数据（JSON 中 `item.status === "approved"`）
- `getStatus()` 函数：SRDPM `item.status==="approved"` → 返回 `"approved"`；localStorage 中 `"confirming"` 或旧 `"approved"` → 返回 `"confirming"`
- `toggleApproval()` 统一逻辑：`pending→confirming`（标记），`confirming→pending`（撤销）
- `confirmApprovals()` 弹窗明确提示："这些条目在SRDPM系统中仍为待审核状态"

### 5.3 关键 JS 函数（在 build_multi_month_dashboard.py 中生成）

| 函数 | 功能 |
|------|------|
| `getStatus(catKey, idx)` | 获取条目状态：先查SRDPM真实status，再查localStorage |
| `toggleApproval(catKey, idx)` | 切换 pending↔confirming，调 `renderCategoryNav()` + `switchTab(catKey)` |
| `switchTab(catKey)` | 切换标签页，直接传catKey参数（不依赖DOM查找active） |
| `confirmApprovals()` | 确认所有confirming条目，弹窗提示SRDPM未真实审批 |
| `renderCategoryNav()` | 渲染左侧导航栏，区分"已标记X · 待审核Y" |
| `migrateManualState()` | 迁移旧localStorage：所有分类的"approved"降级为"confirming" |
| `updatePendingCount()` | 统计底部状态栏，显示各分类待审核/已标记数量 |

### 5.4 JS 常量


## 六、本对话完成的修复（2026-07-10 第四轮）

### 修复的两个核心问题

#### 问题1：点击审核按钮后 UI 不即时更新
- **根因**：`toggleApproval()` 先调 `renderCategoryNav()` 重建导航栏DOM（active class丢失），然后 `switchTab(document.querySelector(".cat-nav-item.active")...)` 找不到active按钮 → renderPanel 不执行 → UI不刷新
- **修复**：改为 `switchTab(catKey)` 直接传当前分类参数

#### 问题2：看板显示"已审核"但 SRDPM 系统实际未审核
- **根因**：三/四分类的 localStorage 存 `"approved"`，`getStatus()` 读出来就显示"已审核"——但只是本地标记
- **修复**：整个状态系统重构为统一三态（见 5.2），localStorage 不能存 `"approved"`

### 具体代码改动（build_multi_month_dashboard.py）

1. **`toggleApproval()`**：统一所有分类为 `pending↔confirming`，`switchTab(catKey)` 直接传参
2. **`getStatus()`**：SRDPM `item.status==="approved"` → `"approved"`；localStorage `"confirming"` 或 `"approved"` → `"confirming"`
3. **`migrateManualState()`**：遍历 `ALL_CATS`，所有分类旧 `"approved"` → `"confirming"`
4. **`renderTable` 状态列**：三/四 confirming 显示"已标记"（而非"待确认"）
5. **`renderTable` 操作列**：三/四 confirming 显示"撤销标记"按钮（而非"✓已勾选"）
6. **`updatePendingCount()`**：统计 confirming 数量，按钮文案简化
7. **`confirmApprovals()`**：弹窗明确提示 SRDPM 系统中仍为待审核
8. **`renderCategoryNav()`**：三/四显示"已标记X · 待审核Y"，二/五/六显示"已勾选X · 待审核Y"
