# SRDPM 工时助手项目约束

本文件适用于当前目录及其全部子目录。

## 沟通与改动原则

- 默认使用中文沟通，先说明结果和风险，再说明实现细节。
- 优先做小步、可回退的修改，保留现有可用功能；禁止为修一个问题而无依据地重写整条链路。
- 所有代码和数据路径必须相对于项目根目录解析，例如以 `Path(__file__).resolve().parent` 为基准。禁止新增指向 WorkBuddy 旧目录或任意个人绝对路径的硬编码。
- `工时审批看板_多月.html` 是生成物，只能通过 `build_multi_month_dashboard.py` 重新生成，禁止直接手工修改生成 HTML。

## 核心主链

唯一主链为：

1. `fetch_and_audit.py`：拉取数据并生成离线审计结果。
2. `srdpm_archive/YYYY-MM/`：保存本地原始数据和审计结果；该目录是敏感运行数据，不属于源码。
3. `build_multi_month_dashboard.py`：读取审计结果并生成多月看板。
4. `工时审批看板_多月.html`：供本地查看、选择和辅助审核的生成页面。
5. `local_approval_server.py`：仅监听 `127.0.0.1`，同源托管看板并在用户确认后桥接真实审批。

`audit_june.py`、旧看板生成器、API 探测脚本及批量审批脚本均为历史参考，不得重新接入主链，除非先完成安全重构和回归验证。

## 凭据与数据安全

- SRDPM 登录凭据只能从环境变量 `SRDPM_USERNAME` 和 `SRDPM_PASSWORD` 读取。
- 禁止在源码、Markdown、JSON、日志、测试夹具、生成 HTML或聊天中写入、输出或复制真实密码、Cookie、token、PAT、Authorization header 或其他凭据。
- 禁止读取或提交旧凭据 JSON。发现历史凭据时只报告文件路径和风险，不显示值，并建议轮换。
- `raw_data`、`srdpm_archive/`、员工工时内容、审核报告、截图、抓包页面、凭据 JSON 和审批日志只能保存在本机，不得提交版本库。

## 真实审批安全边界

- SRDPM 审批和驳回会修改生产数据，审批通过不可撤回。默认禁止任何测试、构建、页面预览或离线脚本访问网络或调用真实审批/驳回接口。
- 单元测试和 Playwright 回归必须完全离线，使用本地脱敏夹具、mock/fake transport，并在测试中显式阻断外网和 SRDPM 域名请求。
- 禁止运行 `batch_approve_*`、`test_approve_api*`、`try_reject_api*`、`discover_approve_api*`、`verify_approve.py` 等历史危险脚本。
- 真实审批不得由静态 HTML 或浏览器 `localStorage` 直接执行。正式执行层必须同时满足：
  1. 基于稳定业务 ID 生成明确审批清单；
  2. 展示人员、日期、项目、工时、异常结论和准确影响范围；
  3. 用户逐项选择后进行独立的二次确认；
  4. 服务端重新校验允许审批的 ID，支持幂等和失败明细；
  5. 提交后立即从 SRDPM 回读真实状态，只有回读通过才显示“已审批”。
- 未得到用户针对当次具体清单的明确确认，不得调用任何真实审批或驳回接口。
- 页面直批请求只能发送 `month + group_keys`；人员、日期、`user_id` 和审批 ID 必须由服务端从当前归档重建。服务必须保持精确 Host/Origin、同源 Fetch Metadata、随机 CSRF、一次性短期 ticket、单任务执行锁和后台任务轮询，禁止 CORS 与自动重试审批提交。
- 提交调用抛异常时必须先只读回查，不能自动重提；只有逐 ID 回读为“通过”才能标记成功，其余必须区分 `not_attempted` 与 `unknown`。

## 统计与状态语义

- “需人工审核的异常”只统计人工异常分类，不得混入平台信息、正常申报或自动审批候选。
- “自动审批候选”“已选择”“已提交”“SRDPM 已审批”必须是不同状态，禁止用 `localStorage` 的本地标记冒充服务器状态。
- UI 状态必须绑定稳定业务 ID，不得使用数组下标作为持久化键。
- SRDPM 父记录 `id` 是每天变化的审批记录 ID，不是人员 `user_id`；只有明确的 `uid/user_id` 才能用于严格人员匹配。缺失时允许姓名兼容查询，但必须继续执行待审 ID 全等校验；出现同名人员时禁止直批，直到取得稳定人员标识。
- 看板分类数据必须直接来自结构化 JSON/raw 数据，禁止从 Markdown 报告反向解析业务记录。

## 测试与验收

- 每次缺陷修复或行为调整都必须同时新增或更新：
  - `unittest` 单元/集成回归，覆盖规则、分类、计数、稳定 ID、状态转换及审批清单；
  - Playwright 离线页面回归，覆盖标签切换、选择/撤销、汇总同步、刷新恢复、分页和重置。
- 修复前先加入能复现问题的失败测试，修复后运行完整离线测试套件。
- 测试不得依赖当前日期、真实账号、浏览器已有 Cookie、内网可达性或生产数据顺序。
- 至少保留以下回归断言：人工异常数量与平台/正常申报严格分离；输入记录不会因换行或特殊字符丢失；重新排序后已选择状态不会错绑。
- 真实审批执行层的任何变更都必须继续保持“默认离线、显式执行、二次确认、提交前状态全等校验、提交后逐ID回读”这五项约束。
